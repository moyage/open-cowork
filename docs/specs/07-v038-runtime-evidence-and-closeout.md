# v0.3.8 运行证据适配与闭环报告可见性

本规格记录 v0.3.8 的迭代需求基线。v0.3.8 的主线来自演进资料包，目标是让外部执行环境的结果能够标准化进入 open-cowork 的证据链；同时，新增 `CHG-V038-STEP-REPORT-VISIBILITY-AND-FINAL-ROUND-CLOSEOUT` 作为前置治理可见性切片，先补齐步骤报告和整轮闭环报告，再推进运行证据接入。

## 语言原则

- 面向人和团队的文档默认使用中文。
- 命令、文件名、字段名、标识符和协议锚点可以保留英文原文，例如 `ocw evidence append`、`runtime_id`、`FINAL_ROUND_REPORT.md`。
- 后续可以增加语种选择，但同一份人类阅读文档不应中英叙述混杂。

## v0.3.8 总目标

- 定义运行环境画像，描述外部执行环境能做什么、风险是什么、如何产出证据。
- 定义证据适配器契约，让外部执行结果以统一结构进入证据链。
- 定义运行事件接入，让终端、会话、工作台、守护进程等事件可记录但不污染权威事实。
- 提供通用终端 / 文件命令适配能力的首轮可执行边界。
- 让外部执行结果成为可审查证据，而不是聊天声称。
- 补齐步骤报告可见性、步骤耗时和最终整轮闭环报告，保证团队能理解真实执行链。
- 将“完整实现、禁止未经批准降级”纳入 open-cowork 的流程和规范。

## 需求切片

### P0：完整实现与禁止未经批准降级

该切片是所有 v0.3.8 需求和后续 change package 的共同治理前提。

要求：

- 已确认的需求、`scope_in`、任务完成定义和验收标准默认要求完整实现。
- Agent 不得自行把需求降级为最小实现、部分实现、延期实现或后续再补。
- 如需降级、拆分、延期或缩减范围，必须先记录原因、影响、剩余风险和拟调整内容，并获得人的明确批准。
- 未获批准的降级不得作为 Step 6 完成、Step 7 验证通过、Step 8 review approve 或 Step 9 archive 的依据。
- Reviewer 必须把未获批准的范围缩减、验收降级或任务遗漏视为阻塞问题。

### P0：步骤报告可见性与最终闭环报告

对应变更包：`CHG-V038-STEP-REPORT-VISIBILITY-AND-FINAL-ROUND-CLOSEOUT`

该切片是 v0.3.8 的前置治理可见性要求，不替代运行证据适配主线。

要求：

- 每个标准步骤结束、即将进入下一步前，必须生成当前步骤报告。
- Agent 必须向人展示当前步骤报告的人类可读摘要。
- 步骤报告必须包含开始时间、完成时间和耗时。
- 对缺少历史时间源的步骤，必须标记为 `unknown` 或 `not_recorded`，不得伪造耗时。
- 第 9 步归档完成后必须自动生成 `FINAL_ROUND_REPORT.md`。
- 最终闭环报告必须从权威产物聚合，不得只依赖聊天上下文。
- 归档收据文件和最终状态快照文件必须引用最终闭环报告。

最终闭环报告至少包含：

- 初始需求。
- 最终完成态。
- 4 阶段 / 9 步实际经历。
- 每一步负责人、参与者、状态、输入、输出和耗时。
- 整轮总耗时。
- 验证、审查和归档结论。
- 复盘总结。
- 下一轮遗留问题和优化建议。

### P1：运行环境画像

运行环境画像用于表达某个本地或外部执行环境的能力、风险和证据输出方式。

建议落点：

- `.governance/runtime-profiles/*.yaml`
- `ocw runtime profile add`
- `ocw runtime profile list`
- `ocw runtime profile show`

最小字段应覆盖：

- 运行环境标识，例如 `runtime_id`。
- 运行环境类型，例如本地命令行、工作台、个人域 Agent 组合。
- 所属参与者。
- 能力边界，例如是否能改代码、运行命令、联网、恢复会话。
- 风险面，例如是否能修改文件、是否能执行 shell。
- 证据支持能力，例如命令输出、文件变更、测试结果、会话摘要。
- 治理约束，例如必须有执行边界、必须遵守允许路径、不能自审。

规则：

- 运行环境画像只描述能力，不拥有治理权。
- 运行环境不能绕过第 5 步、第 8 步、第 9 步。
- open-cowork 不复制 OMOC、OMC、OMX、Claude Code、Codex、OpenCode 等执行器。

### P1：证据适配器契约

证据适配器契约用于把外部运行环境的执行输出转换成 open-cowork 可审查的证据输入。

建议能力：

- `ocw evidence append --adapter <adapter-id>`
- `ocw adapter validate-output <file>`
- `adapter-output.schema.yaml`

适配器输入至少包括：

- 执行边界路径。
- 变更编号。
- 允许路径。
- 验证命令。
- 证据写入目标。
- 运行环境画像。

适配器输出至少包括：

- 变更编号。
- 步骤。
- 运行环境标识。
- 执行开始和完成时间。
- 执行状态。
- 变更文件。
- 运行命令。
- 产物引用。
- 验证结果。
- 风险和后续事项。
- 执行边界引用。
- 证据写入目标。

权威性分级必须结构化标注：

- `trace_only`：运行轨迹，仅供参考。
- `evidence_input`：可进入证据链，但仍需验证 / 审查。
- `verification_result`：可作为验证输入。
- `review_decision`：只有合格审查者才能写入。
- `archive_fact`：只能由归档流程生成。

### P1：运行事件接入

运行事件接入用于吸收外部会话、终端、守护进程、工作台的事件，但不让这些事件成为治理权威事实。

建议落点：

- `ocw runtime-event append`
- `.governance/changes/<change-id>/evidence/runtime-events/**`

事件类型至少覆盖：

- 会话开始。
- 会话恢复。
- 命令派发。
- 命令完成。
- 文件变更。
- 测试完成。
- 运行错误。
- 需要人介入。
- 审查者调用开始。
- 审查者调用完成。

规则：

- 运行事件默认只是轨迹记录。
- 运行事件可以作为证据输入，但不能替代验证、审查或归档。
- 如果运行事件与治理产物冲突，最终报告必须显式标记冲突，并以治理产物为准。

### P1：通用终端 / 文件命令适配

v0.3.8 应至少提供一个通用适配示例，用于把本地命令、终端会话或文件产物转成标准证据输入；示例范围必须与已批准验收标准一致，不得未经批准降级。

建议覆盖：

- 命令输出摘要。
- exit code。
- stdout / stderr 引用。
- 文件变更清单。
- 测试结果。
- 运行环境来源。
- authority 标记。

该能力用于建立薄桥接，不用于把 open-cowork 做成通用运行时。

### P1：审查报告中的运行证据来源

审查报告必须能展示运行证据来源与完整性。

至少回答：

- 证据来自哪个运行环境。
- 证据是轨迹记录、证据输入、验证结果还是审查结论。
- 是否满足执行边界中的证据要求。
- 是否存在运行事件与治理产物的冲突。
- 审查者是否独立于执行者。

### P1：v0.3.7 审计遗留项

v0.3.8 还必须接收 v0.3.7 原始切片审计后的追加项。这些追加项不替代运行证据适配主线，而是补齐团队采用和上下文压缩能力中尚未完全落地的部分。

追加要求：

- 接手资料读取层级必须真实区分最小、标准、深度。
- 第 9 步归档必须引用最终接手资料索引和接手摘要。
- 大型资料任务必须能生成来源索引，并记录分批阅读状态。
- evidence 目录必须能生成证据索引，并被接手资料索引引用。
- `profile apply` 必须支持预览或更明确的安全覆盖策略。
- 成员职责边界必须提供 human_agent_pair、reviewer、runtime owner 的模板或校验能力。
- 大量资料阅读模式必须有压缩检查点产物。
- 接手摘要必须有建议长度上限和防止完整历史复写的测试。

## 非目标

- 不深度实现 OMOC、OMC、OMX、Claude Code、Codex、OpenCode 等执行器。
- 不默认启用钩子、MCP、后台任务、远程频道或定时任务。
- 不引入 GUI、desktop、web platform 或中心化服务。
- 不把外部终端会话、运行环境状态或聊天记录当成治理事实。
- 不让适配器输出直接成为完成结论。
- 不要求人学习 schema 字段、CLI 命令或内部文件路径。
- 不回填历史归档的真实耗时；缺失值只能标记为 `unknown` 或 `not_recorded`。

## 完成门

- 适配器输出可以被校验。
- 至少一个通用命令 / 终端证据适配能力可用。
- 运行事件不污染权威事实。
- 审查报告能展示运行证据来源。
- v0.3.7 审计遗留项已被覆盖；如需降级或移出本轮，必须有人的明确批准记录。
- 未发现未经批准的最小实现、部分实现、延期实现或验收降级。
- 步骤报告结构和 Markdown 输出包含时间字段。
- 进入下一步前可以验证当前步骤报告已生成。
- 归档后自动生成 `FINAL_ROUND_REPORT.md`。
- 归档收据和最终状态快照引用最终闭环报告。
- 全量测试通过：`python3 -m unittest discover -s tests`。
