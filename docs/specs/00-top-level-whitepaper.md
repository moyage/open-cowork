# open-cowork 顶层白皮书

## 1. 文档目的

本白皮书用于重新定义 `open-cowork` 的顶层身份、问题空间、能力边界与长期方向。

它不是实现说明，也不是单轮迭代计划。
它回答的是更上层的问题：

- 这个项目为什么存在；
- 它服务的是哪一种时代背景与协作现实；
- 它要解决的根本问题是什么；
- 它与其他上层或相邻系统的边界是什么；
- 后续持续迭代应围绕什么主轴展开。

## 2. 时代背景：从单一智能走向复杂协作系统

AI 发展正在经历一个关键转向：

- 从“让模型更聪明”
- 走向“让系统在模型不稳定的前提下仍然可控、可恢复、可验证、可协作”

在这一阶段，真正的难点已经不再只是：

- 模型会不会写代码；
- Agent 会不会调工具；
- 单一执行体能不能多做几步。

真正的难点变成了：

- 如何承接复杂任务的长链路推进；
- 如何在上下文不稳定、角色可变化的条件下保持连续性；
- 如何让不同人、不同 Agent、不同 AI Coding 工具在不统一底层栈的前提下协同工作；
- 如何让结果从“像是完成”变成“可验证完成”；
- 如何让复杂任务的推进方式更接近工程系统，而不是更像一段很长的聊天。

`open-cowork` 就是在这个背景下成立的。

## 3. 核心判断：AI 时代的竞争点正在转移

如果每个人都能在个人域中借助大模型、Agent 与 AI Coding 工具快速放大能力，那么未来的重要差异化就不只在于：

- 谁是更强的单兵；
- 谁的个人工具链更顺手；
- 谁能更快完成一个局部任务。

更关键的差异化，会转移到：

- 能否把多个“超级个体”有效组织起来；
- 能否在单一个人域无法独立完成的复杂任务中形成高质量协同；
- 能否让每个强个体各司其职、边界清晰、灵活变动，但仍能通过一套协议与框架稳定协作；
- 能否把个人域能力上升为团队级、组织级的持续资产。

这正是 `open-cowork` 试图回答的问题。

## 4. open-cowork 的顶层定义

`open-cowork` 是一个：

> 面向个人域与团队协作的复杂协作底层框架与协议体系，用于把多个“超级个体”在复杂项目中的工作组织为可分工、可验证、可协同、可持续迭代的受控协作过程。

它不是为某一种工具、某一个 Agent、某一种工作流范式而生。
它更像一个协作底座：

- 上面可以接不同的 Agent；
- 可以接不同 AI Coding 工具；
- 可以吸收不同工程范式；
- 也可以被不同团队按自己的节奏和强度采用。

## 5. 它真正要解决的问题

`open-cowork` 关注的不是某个单点效率问题，而是一整类复杂协作问题：

### 5.1 模糊输入如何变成可推进的工作单元

复杂任务往往起于不完整、不稳定、带噪音的输入。
如果没有统一结构，执行很快就会漂移。

### 5.2 多个强个体如何不互相抵消，而是形成协同

单体能力变强，并不自动意味着团队协作会更顺。
缺少协议和边界时，强执行者之间同样会相互覆盖、重复、断链或失焦。

### 5.3 复杂协作如何被验证，而不是被声称

如果没有结构化 evidence、verify、review 与 archive，复杂项目很容易停留在“已经做了”的口头表述层。

### 5.4 项目如何在中断、压缩、换人之后继续推进

复杂任务不是一次对话。
它需要 handoff、owner transfer、increment、状态语义和恢复入口。

### 5.5 人和团队如何在复杂协作中看见重点

如果系统只对 Agent 友好、不对人友好，就会出现盲盒式推进、文档爆炸、重点失焦和审阅负担失控。

## 6. 设计哲学：方法融合，而不是标签自限

`open-cowork` 不应被某一个方法学标签绑死。

Harness、workflow、runtime、SoD、docs-as-code 等范式，各自都在回答复杂协作中的一部分问题。
因此 `open-cowork` 的原则不是选边站，而是：

- 只要有助于高质量、受控、可恢复、可协作地完成复杂任务，就可以吸收；
- 方法是手段，不是身份；
- 项目的定义，应由它所解决的复杂协作问题决定，而不是由它使用了哪组术语决定。

因此，“治理运行时”可以作为次级能力描述存在，但不应再成为唯一主身份。

## 7. 统一协议，高于统一工具

`open-cowork` 的核心判断之一是：

> 个人域可以异构，但团队协作协议必须尽可能统一。

这里的“统一”，不是统一每个人的模型、IDE、Agent 或命令行。
真正需要统一的是：

- 任务入口
- 阶段语义
- 中间产物结构
- 验证与审查定义
- 风险升级与人工决策点
- 交接、归档与下一轮接续机制

这也是它能够低侵入接入不同个人域的基础。

## 7.1 最小核心协议面

如果 `open-cowork` 要从“方向正确”走向“基线成立”，至少必须先把一组最小核心协议面做实。

这组协议面不是全部能力清单，而是项目在第一阶段能够成立的最小内核：

- 结构化任务入口与 change package
- execution contract 与角色绑定
- evidence / verify / review / archive 闭环
- 最小 continuity packet：至少包含 handoff、当前状态、下一判断点
- 面向人的默认状态面：至少包含 4 阶段视图、当前 owner、waiting-on、next decision

只有当这组最小协议面成立时，`open-cowork` 才能真正体现“复杂协作框架与协议体系”的身份。
在此之外的增强能力，可以继续迭代，但不应反过来遮蔽这个最小核心。

## 8. 能力边界

`open-cowork` 当前的主边界应保持清晰：

### 8.1 它要做的

- 项目级、任务级的复杂协作组织
- 结构化 change 与执行约束
- evidence / verify / review / archive 闭环
- continuity primitives
- 面向不同 Agent / 工具的协议接入
- 面向人的阶段视图与压缩阅读入口

### 8.2 它当前不承担的

- 生态级项目组合治理
- 跨项目优先级仲裁
- 组织级战略编排
- 单一大一统 AI coding runtime
- 重型中心化协作平台

如果需要处理生态级、多项目、多阶段的治理问题，应由独立上层项目承接。

## 9. 与相邻系统的关系

### 9.1 与个人域工具链的关系

`open-cowork` 不替代个人域。
它允许每个人继续使用自己熟悉的 Agent、模型、IDE 与命令体系。

### 9.2 与上层生态治理项目的关系

`open-cowork` 是项目级复杂协作底座。
如果未来需要生态级治理、组合视图、项目间升级与战略同步，应在其上层建立独立项目，而不是把这些能力直接塞回 `open-cowork`。

### 9.3 与可视化或产品壳的关系

`open-cowork` 可以有 CLI、TUI、Dashboard 等外壳，但这些外壳不应反向定义它的本体。
本体首先是协作框架与协议体系，其次才是具体交互形态。

## 10. 后续迭代主轴

后续迭代不应围绕“再多几个命令”展开，而应围绕以下主轴推进：

1. 把复杂协作主链真正做通。
2. 把边界、状态与事实源真正做硬。
3. 把 handoff、owner transfer、increment、sync 等连续性原语变成标准能力。
4. 把机器友好的运行时结构翻译成人和团队也能理解的默认体验层。
5. 在不牺牲低侵入性的前提下，补强对不同 Agent / 工具链的协议适配能力。

## 11. 结论

`open-cowork` 不是一个狭义的“治理运行时”项目，也不是一个为了包装而包装的协作产品。

它更准确的定位是：

> 在 AI 时代，为多个“超级个体”协同完成复杂项目而构建的复杂协作底层框架与协议体系。

接下来一切文档、设计与实现，都应围绕这个定位持续收敛，而不是再次漂移回某个更窄的局部定义。
