## 1. 候选主题

**Default-visible and Enforceable Human Gates**

目标：把 v0.2.7 已引入的 human-control baseline 从“可选产物 / 可记录事实”升级为“默认可见 / 默认接入命令流 / 默认不可静默绕过”的协作治理体验。

## 2. 问题陈述

v0.2.7 已经补齐 participants、intent、step report 和 current-state sync 的最小能力。外部 dogfood 显示方向正确，但仍有三个根问题：

1. **可见性不是默认体验。** Step reports 需要手动生成，Step 1-4 容易缺失，CLI 推进时没有自然向人展示当前步骤、owner、输入、输出、完成标准和下一步进入标准。
2. **参与者配置没有真正进入 step matrix。** 自定义 Agent 会进入 participants 列表，但不会自动或显式映射到 9 步 owner / reviewer / assistants。
3. **human gate 没有执行约束。** intent confirmation、Step 5 approval、Step 8 review gate 和 Step 9 archive gate 仍主要是展示和记录，`run` / `review` / `archive` 尚未消费这些 gate。

因此，v0.2.8 应避免扩展成大而全的团队协作系统，优先把“人类控制基线”接入默认执行路径。

## 3. 候选范围

### R-001：Step reports as default command output

优先级：P0

让 4 阶段 / 9 步成为默认可见流程，而不是可选文件。

候选改动：

- `intent capture` 生成或刷新 Step 1 / Step 2 report。
- `change prepare` 生成或刷新 Step 3 / Step 4 / Step 5 report。
- `run` 生成或刷新 Step 6 report。
- `verify` 生成或刷新 Step 7 report。
- `review` 生成或刷新 Step 8 report。
- `archive` 生成或刷新 Step 9 report。
- 上述命令默认在 CLI 输出中展示 step report 摘要。

验收标准：

- 从 change create 到 archive，归档包中至少包含 Step 1-9 报告。
- 每个推进命令输出当前 step、phase、owner、human gate、done criteria、next-entry criteria 和 recommended next action。
- Agent playbook 明确要求 Agent 将 step report 摘要转述给人，而不是只汇报命令成功。

### R-002：Explicit step participant mapping

优先级：P0

让人能把真实个人域 / 团队域参与者绑定到每一步，而不是只生成默认占位矩阵。

候选命令形态：

```bash
ocw participants setup --profile personal \
  --participant coding-agent:executor \
  --participant verification-agent:verifier \
  --participant review-agent:reviewer \
  --step-owner 6=coding-agent \
  --step-owner 7=verification-agent \
  --step-reviewer 8=review-agent
```

候选改动：

- 支持 `--step-owner <step>=<participant>`。
- 支持 `--step-assistant <step>=<participant>`，可重复。
- 支持 `--step-reviewer <step>=<participant>`。
- 输出未分配 participant warning。
- `change prepare` 不得静默覆盖已有 participants bindings。

验收标准：

- 自定义 Agent 出现在 9 步 owner matrix 的对应步骤中。
- `bindings.yaml` 保留 `participants_profile_ref` 和 step matrix。
- 先 setup 后 prepare 不会丢失人的参与者配置；如需覆盖，必须显式 force。

### R-003：Human gate approval records

优先级：P0

为 human gate 增加明确授权记录，并让关键命令消费该记录。

候选命令形态：

```bash
ocw step approve --change-id <change-id> --step 5 --approved-by human-sponsor
ocw step approve --change-id <change-id> --step 8 --approved-by human-sponsor
ocw step approve --change-id <change-id> --step 9 --approved-by human-sponsor
```

候选改动：

- 新增 `.governance/changes/<change-id>/human-gates.yaml`。
- Step 5 approval 是 `run` 的默认前置条件。
- Step 8 approval 或 approved review decision 是 `archive` 的默认前置条件。
- Step 9 approval 用于关闭前人类确认；如果暂不强制，也必须 warning。
- 提供 `--force` 或 `--ack-risk` 时写入风险记录，不能静默绕过。

验收标准：

- 未通过 Step 5 gate 时，`ocw run` 不应静默执行。
- 未通过 Step 8 gate 或 review approve 时，`ocw archive` 不应静默归档。
- gate bypass 必须在 evidence / manifest / archive receipt 中可审计。

### R-004：Intent-contract consistency checks

优先级：P1

防止人确认的 intent 与实际 contract scope 分叉。

候选改动：

- `contract validate` 对比 intent scope 与 contract `scope_in` / `scope_out`。
- `change prepare` 如果发现 confirmed intent 与 contract 草案冲突，应 warning 或阻止覆盖。
- `step report` 展示 intent-contract drift。

验收标准：

- scope、acceptance criteria、risk 至少有基础差异检查。
- 未确认 intent 时，Step 6 readiness 不应为无条件 true；严格模式下应 false。

### R-005：Human-readable recovery for misordered commands

优先级：P1

让 Agent 和人都能从错误输出中知道下一步，而不是看到 Python traceback。

候选改动：

- `step report` 捕获 draft / invalid contract 错误。
- `review` 校验 reviewer 与 bindings / participants profile 的一致性，不一致时 warning；严格模式下阻止。
- `adopt --dry-run` 输出 `human_control_baseline_next_actions`。

验收标准：

- `change create` 后直接执行 `step report` 输出人类可读错误和建议命令。
- adoption plan 明确提示 participants、intent、step report / approval 是 v0.2.8 默认流程的一部分。

### R-006：Final state consistency snapshot at archive

优先级：P2

让归档包表达最终 archived 状态，而不是只保留执行中间态快照。

候选改动：

- 状态快照增加 `generated_at` 和 `lifecycle_at_generation`。
- `archive` 生成最终 `STATE_CONSISTENCY_CHECK.yaml` 或 `FINAL_STATE_CONSISTENCY_CHECK.yaml`。
- archive receipt 引用最终状态快照。

验收标准：

- 归档后审计者能直接看出哪些快照是执行中生成，哪些代表最终归档状态。
- `doctor --state-consistency` 对 archived change 能解释最终状态一致性。

## 4. 非目标

v0.2.8 不建议默认包含：

- Dashboard / TUI；
- 云端多人协作服务；
- 企业审批系统；
- 项目管理软件级任务看板；
- 与具体 Agent 平台的深度插件化；
- 重新设计整个 open-cowork 4 阶段 / 9 步模型。

## 5. 推荐切片

### Slice A：Default step visibility

包含 R-001 和 adoption / playbook 输出修订。

价值：让人默认看见 9 步流程，不再依赖 Agent 主动读文件。

### Slice B：Participant mapping and preservation

包含 R-002，以及 `change prepare` 保留已有 bindings。

价值：让个人域真实 Agent 进入 owner matrix，解决 v0.2.7 最大困惑点。

### Slice C：Enforced gates and consistency checks

包含 R-003、R-004、R-005 的最小强约束。

价值：让 human gate 从展示型约束变成执行型约束。

### Slice D：Archive audit polish

包含 R-006。

价值：修复归档审计时序语义，降低维护者解释成本。

## 6. 推荐完成定义

v0.2.8 至少应满足：

1. 一轮标准 change 从 intent 到 archive 自动产生 Step 1-9 报告，并在命令输出中显示摘要。
2. 人可以把真实 Agent 显式绑定到 step owner / assistant / reviewer，且 `change prepare` 不会静默覆盖。
3. 没有 Step 5 human gate approval 时，`run` 不会静默执行。
4. 没有 approved review / Step 8 gate 时，`archive` 不会静默关闭。
5. intent 与 contract scope drift 能被检查和报告。
6. draft / 顺序错误 / reviewer 不一致等问题输出人类可读恢复建议。
7. 归档包包含最终状态一致性快照或等价最终审计记录。

## 7. 待维护者确认的问题

1. v0.2.8 是否采用 strong gate 默认阻止，还是先采用 warning + explicit force？
2. `intent confirm` / `step approve` 的 `--approved-by` 是否需要区别 human actor 与 agent actor？
3. Step 9 human gate 是否在 v0.2.8 强制，还是先记录为 archive warning？
4. participant mapping 是否需要交互式向导，还是先用显式 CLI 参数和 Agent playbook 完成？
5. v0.2.8 是否继续作为个人域优先，团队域仅保证模型兼容？
