# open-cowork 人类协作体验反馈与设计方向

## 1. 目的

本文件用于固化一组来自团队成员的初步体验反馈，并将其转化为后续迭代的明确设计方向。

这些反馈不否定 `open-cowork` 的复杂协作主轴，而是指出：

- 当前底层协作骨架对 Agent 编排友好；
- 但对人和团队的默认呈现、默认阅读路径、默认进度感知仍不够友好。

## 2. 原始反馈的核心问题

### F1. 步骤过多，重点不清

当前底层 9 步虽然便于精细约束和 Agent 编排，但从人的体验看：

- 默认看到的步骤过多；
- 每一步在做什么不够清晰；
- owner、参与者、所需能力、人是否需要介入不够显式；
- 默认汇报更像“流程在推进”，而不是“项目在推进”。

### F2. 产出文档过多，导致阅读与管理心智负担过重

当前每轮会生成较多文档。

这带来的问题不是“结构化产物不该存在”，而是：

- 人默认需要面对的文档过多；
- 下一轮的默认输入也容易爆炸；
- 人和团队容易把注意力放在维护文档上，而不是维护项目判断与推进重点上。

### F3. 整体流程可见性不足，存在盲盒感

当前实践中容易出现以下感受：

- 不知道当前阶段在整体流程中的位置；
- 不知道后面还有哪些步骤；
- 执行中的大部分时间只能等待，或主动追问进度；
- 对当前等待谁、下一步是什么、何时需要人介入不够清楚。

### F4. 需要把传统项目语义映射到底层骨架上

只要仍有人和团队参与，就仍会自然地用一些传统开发语义理解协作过程，例如：

- 需求澄清
- 方案设计
- 开发与联调
- 测试与修正
- 评审与验收
- 归档与复盘

因此，`open-cowork` 需要给出：

- 底层 9 步如何被人类默认理解；
- 人默认看到的 4 阶段如何与项目推进语义对应。

## 3. 设计判断

### D1. 保留底层 9 步，但不默认让人直接消费 9 步

底层 9 步仍有必要保留，因为它服务于：

- 执行约束
- gate 控制
- verify / review / archive 不可绕过
- Agent 编排的细粒度约束

但人默认应消费更高层的 4 阶段视图，而不是直接盯着全部 9 步。

### D2. 默认汇报必须从“流程中心”切换到“项目中心”

默认汇报应优先回答：

- 本轮目标是什么；
- 当前阶段已形成了哪些项目结论；
- 当前阻塞是什么；
- 下一步将推进哪些项目内容；
- 哪些地方需要人决策。

而不是优先汇报：

- 当前进入了哪个 step；
- 流程本身完成到了哪里。

### D3. 不应简单减少结构化产物，而应分层设计产物

机器、审计层和 continuity 需要结构化产物，这一层可以相对细。

但人默认需要处理的文档数量必须被强力压缩。

因此后续需要区分：

- runtime artifacts
- operator-facing pack
- archive package

### D4. 进度可见性必须成为一等能力

后续 `open-cowork` 必须能让人随时知道：

- 现在在哪个阶段；
- 已完成什么；
- 还剩什么；
- 在等谁；
- 下一步是什么；
- 什么时候需要人介入。

### D5. 人类体验层不是尾盘包装，而是第一版成立条件的一部分

如果第一版只补底层闭环，不同时补最小人类状态面，那么团队仍然会继续感受到：

- 流程存在，但项目难以跟进；
- 产物存在，但重点难以抓住；
- Agent 能跑，但人接不住。

因此，人类体验层至少要以“最小成立面”的方式进入第一里程碑，而不是排到最后再补。

## 4. 推荐的人类默认视图：4 阶段

### Phase 1. 定义与对齐

映射：

- Step 1 输入接入与问题定界
- Step 2 意图澄清与范围定义

### Phase 2. 方案与准备

映射：

- Step 3 方案塑形
- Step 4 变更包组装
- Step 5 角色绑定与执行准备

### Phase 3. 执行与验证

映射：

- Step 6 隔离执行
- Step 7 验证与纠偏

### Phase 4. 审查与收束

映射：

- Step 8 审查与决策
- Step 9 归档与维护状态更新

## 5. 与传统项目推进语义的参考映射

`open-cowork` 不应简单复制传统开发流程，但应保留其对人仍然有效的治理语义。

参考映射如下：

- 需求澄清 / framing / scoping -> Phase 1 / Step 1-2
- 方案设计 / 任务拆分 / readiness -> Phase 2 / Step 3-5
- 开发 / 联调 / 测试 / 修正 -> Phase 3 / Step 6-7
- review / acceptance / closeout / retrospective -> Phase 4 / Step 8-9

说明：

- 这是语义映射，不是线性瀑布式复刻；
- AI 时代这些活动的边界可能更频繁往返；
- 但它们仍然是人和团队理解项目推进的重要锚点。

## 6. 新增需求：Human / Team Experience Layer

### HX1. 4 阶段默认视图 + 9 步展开视图

默认对人展示 Phase 视图；需要时再展开到底层 Step 视图。

### HX2. 每一步的 owner / participants / capability / human gate 显式化

每一步至少应回答：

- 谁负责
- 谁参与
- 需要什么能力
- 是否需要人介入

### HX3. 项目中心汇报替代流程中心汇报

默认汇报应聚焦项目内容，而不是流程自述。

### HX4. operator-facing pack 压缩机制

默认给人阅读的输入与输出应压缩为极少数核心文档，而不是直接暴露全部 runtime artifacts。

### HX5. progress snapshot / waiting-on / next-decision visibility

需要一个稳定的状态快照，至少显式包含：

- current_phase
- current_step
- completed_steps
- remaining_steps
- current_owner
- waiting_on
- next_decision_point
- human_intervention_required

### HX6. 传统项目语义到 4 阶段 / 9 步的映射说明

帮助人和团队快速建立理解。

### HX7. closeout package 的人类阅读层瘦身

closeout package 可以保持完整，但必须有默认阅读入口和简明摘要。

### HX8. 机器产物与人类产物分层规范

必须正式区分：

- runtime artifacts
- operator pack
- archive package

## 7. 对后续实现与里程碑的直接约束

Human / Team Experience Layer 不应被视为次要包装工作。

其优先级至少应达到：

- `HX1` ~ `HX5`：P0 / P1
- `HX6` ~ `HX8`：P1

对应到版本策略：

- `Milestone 1` 必须纳入最小人类状态面
- `Milestone 2` 再继续补足更完整的人类体验层

其中 `Milestone 1` 至少应落下：

- 默认 4 阶段视图
- current phase / owner / waiting-on / next decision
- 项目中心汇报
- 最小 operator-facing reading pack

原因：

- 这些问题已经直接影响团队是否愿意持续使用这套框架；
- 它们不只是“UI 优化”，而是复杂协作底座是否能被人真正接住的问题。

## 8. 一句话结论

`open-cowork` 可以继续保留底层 9 步骨架，但必须在第一版就提供最小人类状态面，并在默认体验层提供 4 阶段视图、项目中心汇报、压缩阅读包与可感知的进度快照；否则它虽然能被 Agent 使用，却仍难以成为团队愿意持续实践的复杂协作底座。
