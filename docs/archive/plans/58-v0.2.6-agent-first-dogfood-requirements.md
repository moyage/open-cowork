# V0.2.6 迭代需求：Agent-first Dogfood 闭环修复

日期：2026-04-24
来源：`docs/archive/reports/2026-04-24-v025-agent-first-dogfood-findings.md`
基线版本：`v0.2.5`
目标版本：`v0.2.6`
需求性质：dogfood 发现纳入迭代需求

## 1. 背景

V0.2.5 已经把 open-cowork 的默认入口从 CLI-first 调整为 Agent-first：人可以在个人域 Agent 中说“安装 open-cowork，并在当前项目中实施这套协同治理框架”，由 Agent 负责安装、初始化、准备 change、维护状态和汇报进展。

本次 dogfood 以“新的团队成员个人域 Agent 在当前 open-cowork 项目中实施 open-cowork 框架规范和流程”为模拟场景，验证 V0.2.5 是否已经能支撑真实 Agent-first 采用闭环。

结论：V0.2.5 的入口叙事已经转向正确方向，但实现层仍存在多处 CLI 时代遗留断点：运行态迁移、active change 生命周期、contract/runtime 一致性、current-state/digest 投影刷新、evidence 记录和 gitignore 策略都需要在 V0.2.6 先补齐。否则团队成员试用时会继续感受到“说是一句话，实际还是命令和状态迷宫”。

## 2. V0.2.6 目标

V0.2.6 的目标不是扩大功能面，而是修复 V0.2.5 Agent-first 采用闭环中的 P0/P1 断点，让 open-cowork 的第一轮真实个人域试用满足下面标准：

1. 人可以用一句自然语言触发 Agent 采用 open-cowork。
2. Agent 可以自动完成已有项目的状态识别、参数推导、change 准备、handoff pack 生成和状态汇报。
3. contract 是真实执行边界，runtime 不再使用与 contract 冲突的硬编码边界。
4. current-state、status、digest、runtime projection 在 run/verify/review/archive 后保持一致。
5. 已有 active change 未关闭时，框架不会静默覆盖或切换，而是提供明确生命周期策略。
6. 团队成员 dogfood 后不会被未跟踪 runtime 文件和不清楚的 `.governance` 提交策略困住。

## 3. 范围原则

V0.2.6 应聚焦“Agent-first 采用闭环修复”，不进入完整 V0.3 能力扩展。

本轮应做：

- 修复阻断 Agent-first dogfood 的 P0/P1 问题。
- 保持 CLI 存在，但输出改为服务 Agent，而不是让人手动学习。
- 增加必要测试，确保 contract、status、digest 和 evidence 不再漂移。
- 明确 `.governance` 运行态文件的提交/忽略策略。

本轮不做：

- 不做完整 GUI / Dashboard / TUI。
- 不做企业级审批流。
- 不做所有 Agent / IDE 的专用 adapter。
- 不做大规模重构或 V0.3 协议扩展。

## 5. 详细需求

### R-001：已有项目 Agent-first 迁移

问题：

- V0.2.5 发布后，open-cowork 本仓库已有 `.governance/` 仍停留在旧 draft active change。
- 已有项目不会自动出现 `.governance/AGENTS.md`、`.governance/current-state.md`、`.governance/agent-playbook.md`。

需求：

- 提供命令或能力用于刷新已有项目的 Agent-first handoff pack。
- 建议入口：`ocw onboard --refresh-agent-pack` 或 `ocw migrate agent-pack`。
- 对已有 active change，应基于当前真实 change 生成 handoff pack；如果没有 active change，应生成 idle 状态说明。

验收标准：

- 在已有 `.governance/` 项目上执行刷新命令后，生成 `.governance/AGENTS.md`、`.governance/current-state.md`、`.governance/agent-playbook.md`。
- 文件内容引用真实 current change，而不是固定 demo change。
- 没有 active change 时，current-state 清楚说明当前 idle，并给出创建/采用 change 的下一步。

### R-002：active change 生命周期保护

问题：

- 已有 active change 未关闭时，`ocw pilot` 会静默创建并切换到新 change。
- 旧 change 留在 index 中，但没有 superseded / abandoned / stale 标记。

需求：

- `pilot` / `change prepare` 检测到已有未归档 active change 时，不得默认静默切换。
- 提供明确策略：continue current / supersede old / abandon old / archive first / force。
- 策略应写入 changes-index 或对应 lifecycle 文件。

验收标准：

- 默认执行时，已有 active change 会触发清晰提示或机器可读错误。
- Agent 可用非交互参数指定策略，例如 `--active-change-policy supersede`。
- 被 supersede / abandon 的 change 在 status/digest 中不再被当作普通 draft。

### R-003：`ocw run` 使用 contract 真实写边界

问题：

- contract 允许 `docs/**`，但 `ocw run --modified docs/x.md` 被硬编码边界拒绝。
- 这破坏“contract 是执行事实”的核心语义。

需求：

- `ocw run` 从 active contract 读取 `scope_in` / `scope_out`。
- contract 存在时，不再使用硬编码 `allowed_write_scope=[".governance/**", "src/**", "tests/**"]` 覆盖 contract。
- fallback 只能用于缺 contract 的 draft 友好错误。

验收标准：

- contract 允许 `docs/**` 时，`ocw run --modified docs/x.md` 通过。
- contract 未允许 `docs/**` 时，同一命令失败。
- scope_out 匹配时，即使 scope_in 包含上级路径，也必须失败或明确按 scope_out 优先。

### R-004：scope_in / scope_out 冲突校验

问题：

- contract 可以同时包含 `.governance/**` 和 `.governance/index/**`，validation 仍通过。
- Agent 无法判断哪个边界优先。

需求：

- contract validation 检测 scope_in 与 scope_out 的重叠、包含、冲突。
- 如果设计上允许 scope_out 优先，必须在 validation 输出和 contract schema 中明确。
- `change prepare` 应避免生成明显冲突的 scope。

验收标准：

- `.governance/**` + `.governance/index/**` 会给出 warning 或 blocker，具体行为需稳定。
- validation 输出包含冲突路径和处理建议。
- 测试覆盖 glob 包含关系。

### R-005：Agent-first adoption plan 自动推导层

问题：

- 人可以说一句话，但底层仍需要 Agent 自行推导 `change-id`、`title`、`goal`、`scope-in`、`verify-command`。
- 体验质量取决于 Agent 自身能力，而不是框架能力。

需求：

- 提供 `ocw adopt` 或 `ocw apply`，面向 Agent 调用。
- 输入自然语言目标和 target，输出 adoption plan。
- 自动推导候选 change-id、title、scope_in、verify command、active change policy、handoff files。
- 支持 `--dry-run` 和 `--format json`。

验收标准：

- Agent 只需传入 target 和 goal，即可得到可执行 adoption plan。
- plan 会说明需要人确认的真实决策，而不是内部 schema 字段。
- 对 Python 项目可推导 `python3 -m unittest discover -s tests` 或当前项目测试命令候选。

### R-006：status human summary 同步

问题：

- verify 后 status 显示 Step 7，但 project_summary 仍显示 Step 5。
- 人类状态面出现自相矛盾。

需求：

- status human summary 基于 manifest/current step 动态生成。
- 不得在 Step 7 继续显示 Step 5 checklist。
- status 应检测 current-state 是否 stale。

验收标准：

- run 后 status summary 指向 Step 7 verify 或 review entry。
- verify 后 status summary 指向 Step 8 review。
- review 后 status summary 指向 Step 9 archive。

### R-007：digest/runtime projection 刷新

问题：

- verify 后 `ocw status` 显示 Step 7，但 `continuity digest` 仍显示 Step 5。
- digest 读取了旧 runtime projection。

需求：

- 状态变更命令完成后自动刷新 runtime status projection。
- 或 digest 在读取前强制 materialize 最新 projection。
- digest 输出标注 projection 生成时间和来源状态。

验收标准：

- run/verify/review/archive 后立即执行 digest，状态与 manifest/current-change 对齐。
- 测试覆盖 run 后 digest、verify 后 digest。

### R-008：onboard 输出 Agent-first 化

问题：

- `onboard` 输出 `Next commands you can run`，并建议不存在的 `personal-demo`。

需求：

- 改为输出 Agent-first 状态摘要。
- 如果已有 active change，输出真实 active change 下一步。
- 如果没有 active change，输出“建议创建或准备一个 change”，但不引用不存在的固定 ID。
- CLI 命令可作为“Agent internal fallback”附录，不作为主标题。

验收标准：

- onboard 输出不再出现固定 `personal-demo`。
- onboard 输出包含当前项目状态、是否已启用 Agent pack、下一步建议、需要人决策事项。

### R-009：current-state 刷新或 stale 标记

问题：

- `.governance/current-state.md` 是一次性 Step 5 快照，后续不会刷新。

需求：

- 每次状态变更命令刷新 current-state。
- 或 current-state 文件包含 generated_at、source_manifest_status、refresh command，并可被 status 判定 stale。

验收标准：

- verify 后 current-state 显示 Step 7 或下一步 review。
- stale 时 status 输出明确提示，而不是静默使用旧内容。

### R-010：bounded context 进入 handoff pack

问题：

- diagnose-session 能识别 full scan 超预算，但 handoff pack 没有纳入 recommended read set。

需求：

- `.governance/AGENTS.md` 或 current-state 包含 recommended read set。
- 对超预算项目，pilot/onboard 输出提醒不要全仓扫描。

验收标准：

- 在大型项目上生成 handoff pack 后，Agent 能看到最小读取集合。
- diagnose-session 的推荐集合能被 handoff 文件引用或内嵌。

### R-011：手工 evidence artifact 支持

问题：

- 报告类文档无法作为 `run --modified` artifact 记录。
- 当前 change 自身的 evidence 文档也被 reserved boundary 拒绝。

需求：

- 提供 `ocw evidence add` 或 `ocw run --evidence-doc`。
- 允许当前 change 的 `evidence/**` 被记录为 evidence ref。
- 仍禁止修改 contract、bindings、manifest 等 truth source。

验收标准：

- `ocw evidence add --change-id X docs/report.md` 记录 evidence ref。
- `verify` 能读取并展示 evidence doc ref。
- reserved truth source 仍被保护。

### R-012：`.governance` generated files 提交策略

问题：

- V0.2.5 生成 `.governance/AGENTS.md/current-state.md/agent-playbook.md` 后，git status 出现 untracked。
- `.governance/README.md` 说 runtime 产物默认不提交，但新文件不在 gitignore 中。

需求：

- 明确这些文件是应提交协作入口，还是本地 runtime handoff artifact。
- 更新 `.gitignore`、`.governance/README.md`、getting-started 文档。

验收标准：

- dogfood 后 git status 不再出现令人困惑的未跟踪 `.governance` 文件，或文档明确说明为什么要提交。
- 测试或 smoke check 覆盖生成文件与 gitignore 策略。

### R-013：中文默认体验

问题：

- generated playbook 仍有 `Human update template`、`Project goal` 等英文默认字段。

需求：

- 默认模板中文化。
- 保留必要英文锚点给跨 Agent 识别。
- 可通过 locale/profile 控制输出语言。

验收标准：

- 默认中文环境生成的 playbook/current-state 面向人字段为中文。
- 测试覆盖中文模板关键字段。

## 6. 建议切片

### Slice 1：P0 一致性修复

包含：R-003、R-004、R-006、R-007。

目标：先让 contract/runtime/status/digest 不再互相打架。这是后续 Agent-first 自动化的地基。

建议作为 V0.2.6 的第一个实现切片。

### Slice 2：Agent-first 采用闭环

包含：R-001、R-002、R-005、R-008、R-010。

目标：让已有项目和新项目都能通过 Agent-first adoption plan 进入可执行状态，并处理 active change 生命周期。

建议作为 V0.2.6 的第二个实现切片。

### Slice 3：Evidence 与仓库卫生

包含：R-011、R-012、R-013。

目标：修复团队成员 dogfood 后最容易困惑的 artifact 记录、git status 和中文模板体验。

建议作为 V0.2.6 的第三个实现切片，必要时可拆到 V0.2.7，但不应丢失。

## 7. V0.2.6 完成定义

V0.2.6 至少应满足：

1. `ocw run` 以 contract scope 为准。
2. contract validation 能发现 scope 冲突。
3. run/verify 后 status、current-state、digest 一致。
4. 已有 active change 未关闭时，pilot/adopt 不静默覆盖。
5. 已有项目可刷新 Agent-first handoff pack。
6. onboard 输出不再把人带回 CLI-first 命令清单。
7. dogfood 复测不再出现 F-001、F-002、F-004、F-005、F-012、F-013。

## 8. 后续处理建议

- 将本文作为 V0.2.6 change package 的需求输入。
- V0.2.6 不应继续扩大到 Dashboard / TUI / 企业协作，而应先修复 Agent-first 采用闭环的可信度。
- 原始 dogfood 报告保留为 evidence；本文作为整理后的迭代需求基线。
