# V0.2.7 迭代需求输入：个人域 Agent 第一条指令 Dogfood

日期：2026-04-24
来源报告：`docs/archive/reports/2026-04-24-v026-first-step-personal-agent-dogfood-v027-findings.md`
基线版本：`v0.2.5` 代码状态 + `docs/archive/plans/58-v0.2.6-agent-first-dogfood-requirements.md` 作为当前迭代需求来源之一
目标版本：`v0.2.7`
模拟角色：团队成员个人域中的 Codex
用户提供的个人域环境：Openclaw、Hermes agent、Codex、OOSO（Opencode + OMOC + Superpowers + Openspec）

## 1. 需求背景

用户在个人域工作场景中看到 open-cowork 项目后，向 Codex 发出自然语言指令：

```text
根据这个Open-cowork项目（https://github.com/moyage/open-cowork ）的指引，进行安装和配置，在我当前的个人域环境中针对Open-cowork这个项目的迭代实施这套框架的规范和流程
```

本次 dogfood 要验证的不是“熟悉 open-cowork 的维护者能否手动拼命令”，而是“一个团队成员在个人域 Agent 中说出这句话后，Codex 能否依据 open-cowork 指引完成安装、配置、当前迭代接入和流程启动”。

第一步结论：安装可完成，但安装后的 adoption、当前迭代绑定、状态一致性、个人域 agent 组合映射仍然没有闭环。V0.2.7 应把“人一句话 -> Agent adoption plan -> 当前迭代 change/bindings/status”作为重点。

## 3. 详细需求

### V027-R001：Agent adoption 入口

问题：

- 人说一句话后，当前工具没有对应入口。
- `ocw pilot --target . --yes` 会失败并要求 `--goal`。
- change-id、title、goal、scope、verify-command 都需要 Codex 自行推导。

需求：

- 提供面向 Agent 的 `ocw adopt` 或等价能力。
- 输入应支持自然语言 goal、target、可选 source doc。
- 输出 adoption plan，包含候选 change-id、title、scope、verify command、role mapping、active change policy 和需要人确认的真实决策。

验收标准：

- Codex 可用用户原始指令生成 adoption plan，而不是手工拼 `pilot` 参数。
- adoption plan 支持 `--dry-run` 和 `--format json`。
- 缺少必要信息时，工具返回面向 Agent 的澄清项，而不是 argparse usage。

### V027-R002：onboard 真实项目化

问题：

- `onboard` 输出固定 `personal-demo` 下一步命令。
- 当前项目没有 `personal-demo`。

需求：

- onboard 输出基于真实 current change 或明确 idle 状态。
- 不得输出不存在的固定 demo ID。
- 如果当前项目已有 active change，应说明当前 change、状态、下一步和是否存在 stale 风险。

验收标准：

- 任何已有 active change 项目执行 onboard，不会出现 `personal-demo`。
- 没有 active change 时，输出“尚未创建 change”，并建议通过 adopt/pilot 创建。

### V027-R003：状态事实一致性

问题：

- 同一项目中：status 是 Step 7，current-state 是 Step 5，digest 是 Step 5。

需求：

- 状态入口必须由同一状态源或同步刷新机制生成。
- current-state 和 digest projection 不能在 run/verify/review/archive 后滞后。
- 输出中应包含 generated_at/source_status，方便判断 stale。

验收标准：

- verify 后 status/current-state/digest 全部指向 Step 7 或 Step 8 review entry。
- 测试覆盖 run/verify/review 后的三入口一致性。

### V027-R004：安装反馈体验

问题：

- bootstrap 成功后主反馈仍是命令清单。
- 人看不到“Codex 已完成安装，下一步由 Agent 做什么”。

需求：

- bootstrap/update/onboard 输出统一为：安装状态、当前项目状态、Agent 下一步、人是否需要决策、fallback commands。
- fallback commands 不能作为主标题。

验收标准：

- bootstrap 成功后输出 `安装可用` / `命令路径` / `下一步建议` / `是否需要人介入`。
- 人无需理解命令即可知道当前是否 ready。

### V027-R005：仓库卫生诊断

问题：

- 首次执行前已有 untracked `.governance/*` 与 docs 文件。
- 团队成员无法判断哪些应提交、忽略或清理。

需求：

- 提供 `ocw hygiene` / `ocw doctor`。
- 分类输出：runtime generated、pending docs、tracked truth source、ignored artifacts、建议提交、建议忽略、建议清理。

验收标准：

- 当前仓库运行 hygiene 能明确指出 `.governance/AGENTS.md/current-state.md/agent-playbook.md` 的处理策略。
- 对 docs/archive 下新增需求文档，能提示“这是 pending documentation artifact”。

### V027-R006：bounded context 执行约束

问题：

- diagnose-session 给出 recommended read set，但没有进入 handoff/adoption 行为。

需求：

- adoption/onboard/pilot 在发现 full scan 超预算时，自动写入 bounded context 指引。
- `.governance/AGENTS.md` 或 current-state 包含 recommended read set。

验收标准：

- 诊断出超预算后，Agent handoff 文件明确“不要全仓扫描，先读以下文件”。
- Codex 下一步汇报能自然引用 read set。

### V027-R007：个人域 agent inventory 与 role bindings

问题：

- 用户提供 Openclaw、Hermes、Codex、OOSO，但当前 bindings 仍是通用占位。

需求：

- adoption plan 支持记录个人域 agent/tool inventory。
- 根据用户提供的工具组合生成 role mapping 候选：orchestrator、executor、verifier、reviewer、maintainer。
- 写入 bindings 前需要人确认。

验收标准：

- adoption plan 能展示“基于你提供的个人域组合，我建议 ...”。
- bindings 不再只保留 `orchestrator-agent`、`executor-agent` 等无上下文占位。

### V027-R008：source doc 绑定

问题：

- 用户明确指出 `docs/archive/plans/58-v0.2.6-agent-first-dogfood-requirements.md` 是当前迭代需求来源之一。
- 当前 open-cowork 无法把它登记为 change source。

需求：

- `ocw adopt/change prepare` 支持 `--source-doc`。
- intent、requirements、manifest、status、digest 能展示 source docs。

验收标准：

- 生成 change package 后，manifest 或 requirements 明确引用 source doc。
- digest 显示当前 change 的需求来源。

### V027-R009：安装降级文案

问题：

- bootstrap shim fallback 文案容易被误解成失败。

需求：

- 文案改成“已使用本地 shim 安装，可正常使用”。
- 技术细节放入 diagnostics。

验收标准：

- 非 Python 用户看到输出也能理解安装已成功。

## 4. 建议 V0.2.7 切片

### Slice A：第一条指令可承接

包含：V027-R001、V027-R002、V027-R008。

目标：人给一句话和 source doc 后，Agent 能生成 adoption plan 并进入当前迭代 change。

### Slice B：状态可信

包含：V027-R003、V027-R006。

目标：status/current-state/digest/read-set 不再互相打架。

### Slice C：个人域体验

包含：V027-R004、V027-R005、V027-R007、V027-R009。

目标：安装反馈、仓库卫生、个人域 agent 组合和安装文案更像 AI 时代工具，而不是命令行教程。

## 5. V0.2.7 完成定义

V0.2.7 至少应满足：

1. 用户的第一条自然语言指令可以被 Codex 转化为 adoption plan。
2. adoption plan 能引用 `docs/archive/plans/58-v0.2.6-agent-first-dogfood-requirements.md` 作为 source doc。
3. onboard 不再输出不存在的 `personal-demo`。
4. status/current-state/digest 对当前 active change 保持一致。
5. diagnose-session 的 recommended read set 进入 Agent handoff。
6. 个人域 agent inventory 能进入 role binding 候选。
7. bootstrap 成功反馈不再让人误以为安装失败或需要自己继续敲命令。

## 6. 本轮第一步结论

V0.2.7 的核心不是增加更多命令，而是让 open-cowork 真正能承接“团队成员在个人域 Agent 中发出的第一条自然语言采用指令”。

如果这一步不能闭环，后续 change、contract、evidence、review、archive 都会继续依赖 Agent 临场拼接，而不是由 open-cowork 框架提供稳定协议入口。
