## 1. 候选主题

**Human-visible Collaboration Gates**

目标：让 open-cowork 从“Agent 能跑完 governance lifecycle”升级为“人能看见、配置、确认和授权个人域/团队域多参与者协作流程”。

## 2. 问题陈述

v0.2.6 已经能完成 Agent-first adoption closure，但外部 dogfood 反馈显示：

- 人没有感受到自己能配置和控制不同 Agent 在 4 阶段 9 步中的职责；
- 人没有看到明确的意图、需求、范围、验收标准确认关卡；
- 9 步流程缺少逐步报告和推进门槛；
- 归档、快照和 current-state 的人类可读状态仍可能与机器状态不一致；
- CLI 易用性问题会干扰首次采用。

这些问题会让 open-cowork 退化成“单 Agent workflow harness”，与其个人域/团队域/多 Agent 复杂协作治理定位不一致。

## 3. 候选范围

### R-001：Participants / Owner setup

优先级：P0

提供 Agent-first 的参与者配置流程，让人能确认 4 阶段 9 步中每一步的：

- primary owner；
- assistant participants；
- verifier；
- reviewer；
- human gate；
- final decision owner。

候选命令形态：

```bash
ocw participants setup --profile personal
ocw participants setup --profile team
```

候选输出：

- `.governance/participants.yaml`
- `.governance/changes/<change-id>/bindings.yaml` 更新建议
- human-readable step-owner matrix

验收标准：

- adoption / pilot / change prepare 能引用 participants profile。
- 输出中明确展示 9 步 owner matrix。
- review owner 不能与 executor 静默合并。
- human gate 步骤在报告中明确标出。

### R-002：Intent capture and confirmation

优先级：P0

提供意图捕获和确认关卡，避免只用一句 `--goal` 直接进入执行。

候选命令形态：

```bash
ocw intent capture --change-id <change-id>
ocw intent confirm --change-id <change-id>
```

候选内容：

- project intent；
- new requirements；
- optimization items；
- bug fixes；
- scope in；
- scope out；
- acceptance criteria；
- risks；
- open questions；
- human confirmation record。

验收标准：

- 未确认 intent 时，Step 5 / Step 6 输出应提示风险或阻止自动推进。
- `change prepare` 能从 confirmed intent 生成 requirements / design / tasks。
- 人类可读报告中展示“本次真实迭代到底要做什么”。

### R-003：Step report and step gate

优先级：P0

为 4 阶段 9 步提供标准化阶段报告和推进门槛。

候选命令形态：

```bash
ocw step report --change-id <change-id> --step <n>
ocw step advance --change-id <change-id> --from <n> --to <n+1>
```

每步报告应包含：

- current step；
- phase；
- owner；
- participants；
- objective；
- inputs；
- outputs；
- done criteria；
- next-entry criteria；
- blockers；
- human decisions required；
- recommended next action。

验收标准：

- Agent-first playbook 要求 Agent 在进入/完成关键步骤时向人报告 step report 摘要。
- 带 human gate 的步骤默认不能被静默跨越。
- `status` / `current-state.md` 能指向最新 step report。

### R-004：Human-readable and machine-readable state consistency

优先级：P1

修复归档后 `current-state.md` 滞后、verify 快照时序不清等问题。

候选能力：

```bash
ocw status --sync-current-state
ocw doctor --state-consistency
```

验收标准：

- `archive` 后 `.governance/current-state.md` 自动刷新为 idle / archived 状态。
- doctor 能发现 current-state、index、runtime-status、archive receipt 不一致。
- verify 输出包含 `snapshot_generated_at` 与 `lifecycle_at_verify`。

### R-005：CLI first-use ergonomics

优先级：P2

修复首次采用中的低成本易用性问题。

候选修复：

- PATH 检测和 shell 提示；
- `ocw contract validate <change-id>` 位置参数兼容；
- outside write boundary 报错展示允许路径和推荐 evidence 参数；
- adoption / onboard 输出更少要求人记命令名。

验收标准：

- 首次使用的错误提示能直接告诉 Agent 或人下一步怎么修。
- 常见误用有兼容路径或明确纠错示例。

## 4. 非目标

v0.2.7 候选范围不应默认包含：

- Dashboard / TUI；
- 云端多人协作服务；
- 企业级审批流；
- 与具体 Agent 平台深度插件化；
- 完整替代项目管理软件；
- 单 Agent harness 化改造。

## 5. 切片建议

### Slice A：Human control baseline

包含：

- participants setup 最小版本；
- intent capture / confirm 最小版本；
- current-state archive 同步。

价值：先让人能配置“谁参与”和“要做什么”，并修掉归档后误导。

### Slice B：Step visibility and gates

包含：

- step report；
- step advance；
- human gate 标注；
- playbook 汇报规则更新。

价值：让 9 步流程从文件事实变成人能感知的推进过程。

### Slice C：First-use ergonomics

包含：

- PATH doctor；
- `contract validate` 参数兼容；
- run boundary 报错优化；
- verify snapshot metadata。

价值：减少首次采用挫败感和误判。

## 6. 推荐 v0.2.7 完成定义

v0.2.7 至少应满足：

1. 人能在个人域配置多个 Agent 的 9 步 owner / participant / reviewer / human gate。
2. 人能在进入执行前确认本轮真实意图、需求、非目标和验收标准。
3. 每个关键步骤都有可读 step report，Agent 汇报时不再只说命令成功。
4. archive 后 human-readable current-state 与 machine-readable index/runtime/archive 一致。
5. 首次采用中的常见 CLI 误用有明确纠错路径。

## 7. 待维护者确认的问题

1. v0.2.7 是否以 Human-visible Collaboration Gates 为唯一主题？
2. participants profile 是否先做 personal，再扩展 team？
3. intent confirmation 是强制 gate，还是先作为 warning gate？
4. step advance 是否需要真实阻止状态跳转，还是先提供报告和建议？
5. v0.2.7 是否要包含 CLI ergonomics，或拆为 v0.2.7.x patch？
